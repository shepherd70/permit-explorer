# permit-explorer — Audit (2026-06)

**Dimensions run:** Correctness & analytics math · Security (web/DOM injection, query injection, supply-chain) & provenance · Maintainability/duplication-drift · Python/build correctness · JS conventions · Repo/CI/docs hygiene. Skipped as not applicable: regulatory defensibility, statistical-estimator validity, financial-data correctness, citation/methods integrity (this is a public, read-only open-data viewer, not a regulated/money/estimator codebase).

**Method:** read every source file (build.py, both HTML apps, three test harnesses, three CI workflows, README/TASKS/docs); ran `pi_audit.py` (injection/secrets/Unicode) plus a manual git-provenance check; confirmed a green baseline (`python build.py` + `npm test` → 56/56 assertions pass); then fanned out five independent dimension-finders and adversarially verified all 26 candidate findings (25 confirmed, 1 refuted). No code was changed.

## Summary

The repo is in good shape: clean git history, no secrets, a thoughtful test setup (offline harnesses + a self-extracting live-API smoke test), and a live app (`src/city_explorer.html`) that escapes user-derived data carefully. **There are no critical findings.** The two issues worth acting on are both in the *offline* path, which has not kept pace with the live app: (1) the offline dashboard renders untrusted permit free-text into the DOM **without HTML-escaping** — a stored-XSS/HTML-injection bug the live app already solved with `esc()`; and (2) `build.py` will **crash the build** if Calgary ever ships the cost column as plain numbers instead of `$1,234`. Everything else is minor — mostly *drift* between the two intentionally-duplicated apps (formatters, negative-day handling, a mislabelled chart) plus routine hardening (SRI, least-privilege CI token, SHA-pinned actions) and test-hygiene. The recurring theme: the live and offline code paths share logic by copy, and the copies have quietly diverged.

## Critical

_None._

## Major

- **Stored XSS / HTML-injection in the offline dashboard — untrusted permit fields rendered unescaped.** `src/template.html` injects applicant/staff-supplied free-text (`description`, `contractorname`, `originaladdress`, plus `id`/`status`/`cls`/`work`) straight into the DOM with no escaping, at three sinks: table rows (`src/template.html:396-400`, written via `innerHTML` at `:403`), the Leaflet map popup (`src/template.html:321`), and the insights panel (`src/template.html:338`, `innerHTML` at `:366`). `build.py`'s `clean_col()` only collapses whitespace (`build.py:66-70`) and its `</`→`<\/` step (`build.py:131`) only protects the JSON-in-`<script>` parse — neither escapes HTML. A permit description such as `<img src=x onerror=…>` executes when the row/popup renders. The live app already does this correctly — `esc()` at `src/city_explorer.html:449`, applied to every one of these fields at `:985-989` (table) and `:822` (popup) — so this is pure drift in the offline copy. **Impact is bounded** (the artifact is a self-contained offline file with no cookies/auth/secrets and the data is public/read-only), which is why verifiers rated it major rather than critical — but it is an easily-triggered injection in a *shareable* file. **Fix:** add the same `esc()` helper to `template.html`'s script and wrap every untrusted interpolation (escape the description *after* slicing, as the live app does); or HTML-escape the text columns in `build.py`.

- **`build.py` crashes when `EstProjectCost` is inferred as a numeric column.** `build.py:75` does `df["EstProjectCost"].str.replace(r"[\$,]", "", regex=True)`. The `.str` accessor only works on object/string dtype. It works *today* only because every value carries a literal `$` (e.g. `$2,000,000`), forcing object dtype (confirmed against the committed CSV). If a future Calgary export drops the `$`/`,` formatting, or the column is empty, or even *mixed*, pandas infers float64/int64 and `.str` raises `AttributeError: Can only use .str accessor with string values!`, aborting the whole build (reproduced on pandas 2.2.3 for all three cases). The very next line handles `TotalSqFt` the robust way — `pd.to_numeric(df["TotalSqFt"], errors="coerce")` with no `.str` (`build.py:76`) — so the safe pattern is already in the file. This is *latent* (not currently triggered, fails loudly, build-time only — hence a major/minor split among verifiers) but it depends on an upstream format change that is exactly the kind of thing this "ingest the newest CSV" pipeline will eventually meet, and no test exercises `build.py`'s CSV path. **Fix:** `pd.to_numeric(df["EstProjectCost"].astype(str).str.replace(r"[\$,]", "", regex=True), errors="coerce")` (the `astype(str)` convention is already used by `clean_col` at `:69`), or read with `dtype={"EstProjectCost": str}`.

## Minor

### Correctness & metric consistency

- **Processing-speed chart is labelled "avg" but plots a median in detail view.** The card hint and `aria-label` are hard-coded "avg days applied→issued" (`src/city_explorer.html:391`) and never change per mode, but the series is a server **mean** in city view (`avg(date_diff_d…)`, `:638`→`:651`) and a client-side **median** in detail view (`median(g[k].ds)`, `:688`→`:756`). It even contradicts the KPI right above it, which *does* flip its label to "Median days to issue" in detail mode (`:730` vs `:739`). The offline template is internally consistent here (median data + "median" label). **Fix:** make the chart hint/`aria-label` reflect the statistic actually plotted in each mode (update them in `render()`).

- **Negative days-to-issue (issued before applied) handled three different ways.** City aggregates average the raw `date_diff_d` with no `>= 0` guard, so negatives drag the mean down (`src/city_explorer.html:623,638,644`); detail mode clamps each value to 0 (`Math.max(0,…)`, `:673`); `build.py` drops them entirely (`build.py:83-84`). The same record therefore contributes a negative (city KPI/year/choropleth), a 0 (detail), or nothing (offline) to a headline "days to issue" number. (The DTI *histograms* happen to agree — negatives and clamped-0s both land in the "0–1 d" bin.) **Fix:** pick one rule across all three paths — simplest is to exclude negatives everywhere (add `date_diff_d(issueddate,applieddate) >= 0` to the city queries; use `null` instead of clamp-to-0 in detail; `build.py` already drops).

- **Completion-rate denominator treats NULL statuses inconsistently within city view.** The KPI builds `resolved` from the status query, which is filtered `statuscurrent IS NOT NULL` (`src/city_explorer.html:643`, used at `:745-748`), so NULL-status permits are excluded from numerator *and* denominator. The choropleth/community rate instead computes `resolved = n − openn` where `n = count(1)` with no status filter (`:644`, `:655`), counting NULL-status permits in the denominator. The two completion rates can disagree for the same community. (No NULL `statuscurrent` rows exist in the dataset today, so latent.) **Fix:** compute the community `resolved` with a `sum(case(statuscurrent is not null and … ,1,true,0))` mirroring the KPI's non-null definition.

- **City "% new construction" note can silently blank out.** The work query is capped at the top 8 work classes (`:640`); the note reads `'New'` out of that truncated list and shows nothing if it's absent (`:742-743`). A filter that pushes `'New'` below the 8th-ranked class hides the (small but real) percentage. Detail view is unaffected (computes from the full row set, `:732`). **Fix:** derive the `New` count from a dedicated server count rather than the truncated top-8.

- **Detail-mode status list truncated to top-20 while the city status query is unbounded.** Detail builds `s.status` via `top('status',20)` (`:690`); `render()` derives completion-rate inputs by scanning it (`:745-747`). A ≤30k selection with >20 distinct statuses would drop the rarest from the resolved sum. Latent (the dataset has 24 statuses, but the most status-diverse detail-eligible selections peak at 19, and `Completed` can never fall out of the top 20). **Fix:** compute `done`/`resolved` from `this.rows` in detail mode, or drop the slice.

### Duplication drift (live `city_explorer.html` vs offline `template.html`)

- **`fmt$` diverged:** the offline copy (`src/template.html:146`) lacks the live copy's billions (`$…B`) tier and `isNaN` guard (`src/city_explorer.html:445`); a ≥ $1B total renders as e.g. `$3100.00M`. A full-city/multi-year embed realistically exceeds $1B, so the missing tier is triggerable (output is awkward but numerically correct).
- **`fmtN` diverged:** the offline copy (`:147`) omits the live copy's `Math.round()` and `isNaN` guard (`:446`). Latent (current call sites pass integers only).
- **Renovation-pairing scope diverged:** the offline file pairs against the *full* embedded history then filters by the renovation permit's id (`template.html:152-166`, `:302`), so an out-of-year-range new-build can still anchor a pair; the live app recomputes only over in-scope rows (`city_explorer.html:694-703`), so the anchoring new-build must itself be in scope. For the same year filter the two report different lag sets/medians.
- **Negative-DTI handling diverged** (see correctness item above) — `build.py` drops, live-detail clamps to 0.
  - **Fix (all of the above):** extract the shared helpers/constants/algorithms into one source the build can inline into both, or at minimum keep `template.html`'s copies byte-synced to the live versions. This is the root cause behind several findings.

### Security hardening (low likelihood under the read-only/public threat model)

- **`onclick` attribute breakout via a `"` in a community name.** Drill links build inline handlers `onclick="D.drill('${esc(name)…}')"` (`src/city_explorer.html:842`, `:879`); `esc()` escapes `& < >` but **not** `"` (`:449`), so a name containing a double-quote would terminate the attribute. Low likelihood (`communityname` is an official controlled vocabulary), but it's the app's designated escaper being incomplete for the attribute context — and it would also break the link for any legitimate name containing `"`. **Fix:** also escape `"` in `esc()`, or bind the handler programmatically via a `data-` attribute instead of string-built inline JS.

- **CSV export lacks spreadsheet-formula-injection guard.** `csvCell()` RFC-quotes but doesn't neutralize cells starting with `= + - @` (`src/city_explorer.html:1039`); a crafted `description` like `=HYPERLINK("http://evil","x")` becomes a live formula when the exported CSV is opened in Excel/Sheets (`:1046-1047`). Gated by several preconditions and spreadsheet warnings. **Fix:** prefix risky cells with a leading `'` (or tab) in `csvCell`.

- **CDN scripts/styles have no Subresource Integrity.** Chart.js, Leaflet JS/CSS, and the Inter font load via plain `<script>/<link>` with no `integrity=`/`crossorigin=` (`src/city_explorer.html:13-15`; identical `src/template.html:7-9`). A CDN/account compromise would run arbitrary JS; the offline file re-fetches these live, so an old shared dashboard inherits whatever the CDN later serves. **Fix:** add pinned SRI hashes (kept in sync across both files) or self-host. _Note: chart.js's path-unpinned URL must be pinned to the resolved asset before hashing._

### Repo / CI / build hygiene

- **`build.py` embeds 5 per-row fields the offline template never reads.** `type`, `sqft`, `issued`, `completed`, `dtc` are emitted for every record (`build.py:107-123`) but never referenced by `template.html` (whose only sink is `DATA` at `:142`). Measured: dropping them shrinks the embedded JSON ~162 KB / **25%** — meaningful for an artifact whose whole point is a compact, shareable file (`sqft` is null on all rows). **Fix:** drop the unused keys from `load()`, or wire them into the template if intended.

- **`find_csv()` selects newest-by-mtime but `export_date()` labels by filename date.** Selection key (`p.stat().st_mtime`, `build.py:28-31`) and freshness-label key (YYYYMMDD in the filename, `:34-45`) are different signals; a re-downloaded older snapshot, or an mtime reset by `git checkout`/copy, can make the loaded file differ from the newest-*dated* file present. (The displayed date always matches the file actually built, so it's a selection surprise, not a wrong label.) **Fix:** sort by the parsed filename date with mtime only as a fallback/tie-breaker.

- **`requirements.txt` is unpinned (`pandas>=1.5`)** despite `build.py`'s stated byte-for-byte determinism goal (`build.py:149-151`) and the committed `dist/` file. Cross-major pandas date-parsing changes could shift the embedded JSON (nothing in CI enforces the determinism — no `git diff --exit-code dist/`). **Fix:** pin (`pandas==2.2.3`) or bound (`pandas>=2.0,<3.0`).

- **`ci.yml` has no `permissions:` block.** The other two workflows scope the `GITHUB_TOKEN` (`pages.yml:13-16`, `smoke.yml:14-15`) but `ci.yml` doesn't (`.github/workflows/ci.yml:1-33`), inheriting the repo default. The repo default is currently `read` (verified via the API), so no live over-privilege — but an explicit block is self-documenting and survives a future default change. **Fix:** add `permissions:\n  contents: read`.

- **Actions pinned to mutable major tags, not commit SHAs.** All steps use `@v6`/`@v5` across the three workflows. First-party `actions/*` from a trusted publisher and near-zero secret exposure, so hygiene-only. **Fix:** pin to 40-char SHAs with a `# v6` comment and enable Dependabot for `github-actions`.

- **Test-harness hygiene (test-only, no product impact):**
  - `test/harness.js:24` and `test/city_harness.js:59` dereference `html.match(/<script>…/)[1]` with no null guard — an edit that violates the single-bare-`<script>` contract throws an opaque `TypeError` instead of a clear message (`test/smoke.js:31-32` already guards its extracts). **Fix:** guard the match like `smoke.js` does.
  - `test/city_harness.js:47` routes the detail-mode stub on a hardcoded `'$limit'==='30000'` mirroring `DETAIL_THRESHOLD` (`city_explorer.html:437`); changing the constant breaks the stub silently (misleading assertion failure). **Fix:** self-extract `DETAIL_THRESHOLD` from source (as `smoke.js` extracts `API`/`fields`) or match on a high `$limit`.
  - `test/harness.js` pins golden magic numbers tied to the committed build (`:34` count 1,498; `:67` renov band; `:78` cost-hist 1,424). A documented data refresh breaks `npm test` with an opaque message and no "data changed, not a regression" hint. **Fix:** relax exact-equality asserts to invariants, or add a single clearly-commented "update on data refresh" anchor.

## Could not verify / ruled out

- **Detail `Math.round(ms/864e5)` vs SoQL `date_diff_d` off-by-one** — *ruled out (false positive).* Both `applieddate`/`issueddate` are midnight-only timestamps, and an exhaustive scan of 78k date pairs (incl. every Mountain-Time and Lord Howe DST transition) found 0 mismatches; a ±1h DST shift can't cross `Math.round`'s 0.5 boundary. The negative-clamping difference is real, but is captured under the negative-DTI finding above — this specific rounding divergence does not occur.
